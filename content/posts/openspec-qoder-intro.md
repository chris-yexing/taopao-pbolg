在构建这个博客的过程中，我尝试了一种新的开发方式——使用 **OpenSpec** 结合 **VS Code + AI 大模型**进行规范驱动的开发。老实讲，刚开始觉得这套流程有点繁琐，但用下来发现它解决了一个真实问题：AI 总是不理解你到底要做什么。

## 什么是 OpenSpec？

**OpenSpec** 是一种规范驱动开发（Spec-Driven Development）方法论。核心思路是：动手写代码之前，先用结构化文档说清楚要做什么、为什么做、怎么做。

### OpenSpec 的核心组件

| 组件 | 作用 | 文件 |
|------|------|------|
| **Proposal** | 回答"为什么"和"做什么" | `proposal.md` |
| **Design** | 回答"怎么做"，包含目标、决策、风险 | `design.md` |
| **Specs** | 详细的功能规格说明 | `specs/*/spec.md` |
| **Tasks** | 可执行的实施任务列表 | `tasks.md` |

### OpenSpec 的工作流程

```
Proposal（提案）→ Design（设计）→ Specs（规格）→ Tasks（任务）→ Implementation（实现）
```

这种方式强迫你在动手前先想清楚，减少返工，也方便后期回头看自己当时为什么这么决定。

## 什么是 Qoder？

**VS Code + AI 大模型**作为编程助手，与 OpenSpec 配合时能读取规范文档、理解项目需求，生成或修改代码时准确度明显更高。相比随手描述需求，AI 拿到结构化文档后出错少很多。

## 本项目中的实际应用

### 项目背景

这个博客网站是我使用 OpenSpec + Qoder 完成的第一个完整项目。

### 我的 OpenSpec 结构

```
openspec/
└── changes/archive/2026-04-09-personal-article-blog/
    ├── proposal.md       # 为什么需要这个博客
    ├── design.md         # 技术选型和架构决策
    ├── tasks.md          # 具体实施步骤
    └── specs/
        ├── article-display/   # 文章展示功能
        ├── blog-search/       # 搜索功能
        ├── rss-feed/          # RSS 订阅
        └── social-sharing/    # 社交分享
```

### Proposal：明确目标

在 `proposal.md` 中，我定义了：

- **Why**：需要一个简单维护的个人博客平台
- **What**：静态博客，支持文章、标签、搜索、RSS、分享
- **Capabilities**：四个核心功能模块
- **Impact**：新项目，无历史包袱

### Design：做出决策

在 `design.md` 中，我记录了关键决策：

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 静态生成器 | Hugo | 简单、快速、功能齐全 |
| 部署平台 | GitHub Pages | 免费、与 GitHub 集成好 |
| 内容管理 | Markdown + Git | 无需 CMS，直接用编辑器 |
| 主题 | PaperMod | 现代、响应式、功能丰富 |

同时也明确了**不做什么**：不支持动态评论、不使用数据库、不做复杂交互。

### 与 Qoder 协作的过程

1. **初始化阶段**：我向 Qoder 说明要用 OpenSpec 搭建博客，它帮我创建了完整的目录结构
2. **规划阶段**：我撰写 proposal 和 design，Qoder 读取并理解这些文档
3. **实施阶段**：Qoder 根据 tasks.md 执行具体步骤：
   - 初始化 Hugo 站点
   - 添加 PaperMod 主题
   - 配置 config.toml
   - 创建第一篇文章
4. **迭代阶段**：我需要修改时，Qoder 基于已有上下文快速响应

### 实际效果

| 方面 | 体验 |
|------|------|
| **开发效率** | 高，AI 理解规范后执行准确 |
| **代码质量** | 符合规范要求，结构清晰 |
| **可维护性** | 文档齐全，后期修改有依据 |
| **学习成本** | 需要适应"先写文档再写代码"的节奏 |

## 优势与局限

### 优势

说实话，最大的收益不是 AI 执行得更准，而是我自己在写 proposal 和 design 的过程中就想清楚了很多东西。规范文档逼着你做决策、记录理由，事后改需求时也有依据可查，而不是完全靠记忆。对团队协作来说这点尤其有价值——规范是比口头沟通可靠得多的契约。

### 局限

这套方式不适合"边做边想"的探索阶段。需求模糊时强行写规范，写出来的东西大概率是废的。另外小改动也要维护文档，有时确实嫌麻烦。工具依赖也是个问题，没有 AI 加持的话这套流程比普通文档重太多。

## 总结

用一句话概括这套组合的本质：规范是人和 AI 之间的契约，人负责决策，AI 负责执行。

对这个博客这样的小项目，这套方式确实稍微重了点。但它带来的意外收获是——我对自己在做什么、为什么这么做想得更清楚了。对于需要长期维护的项目，我觉得这种"先慢后快"的节奏是值得的。后续还会继续在这上面实践，有新的体会再写。
