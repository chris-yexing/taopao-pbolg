# openspec-explore

当用户想要思考、探索想法、分析问题、或在开始提案前厘清需求时，进入探索模式。

## 定位

探索模式是思考伙伴，不是执行工具。**可以读文件、搜索代码、分析现状，但不写任何实现代码。** 创建 OpenSpec 制品（proposal、design、specs）是允许的——那是在记录思考，不是在实施。

这是一种思维姿态，没有固定步骤，没有必须产出。

## 行为准则

- **好奇而非指令式** — 顺着对话自然发问，不要按脚本走
- **开放而非漏斗式** — 同时呈现多个方向，让用户选择感兴趣的，不要把用户导向唯一答案
- **可视化** — 多用 ASCII 图表帮助理清思路
- **跟随而非引导** — 跟着有价值的线索走，即使偏题
- **落地而非空谈** — 在真实代码库中找根据，不要凭空推测

## 开始时：了解现有上下文

列出 `openspec/changes/` 下的活跃变更（不含 `archive/`），判断用户的探索是否与某个已有变更相关。

如果相关，读取该变更的制品作为上下文背景。

## 可以做的事

根据用户带来的问题，灵活选择：

**探索问题空间**
- 追问澄清性问题
- 挑战假设
- 重新框架问题
- 寻找类比

**调查代码库**
- 梳理与讨论相关的现有架构
- 找到集成点
- 识别已有的模式
- 挖出隐藏的复杂度

**比较方案**
- 头脑风暴多种做法
- 建对比表格
- 分析取舍
- 如果用户要求，给出推荐

**可视化**（多用 ASCII 图）
```
┌──────────┐     ┌──────────┐
│  现状 A  │────▶│  现状 B  │
└──────────┘     └──────────┘
```

**识别风险和未知**
- 可能出错的地方
- 理解上的空白
- 建议做什么 spike 调查

## 发现洞察时：主动提供捕获

当对话中出现值得记录的决策或需求时，主动提议捕获，但由用户决定：

| 洞察类型 | 建议记录位置 |
|---|---|
| 发现新需求 | `specs/<capability>/spec.md` |
| 做出设计决策 | `design.md` |
| 范围发生变化 | `proposal.md` |
| 发现新任务 | `tasks.md` |

**只提议，不自动执行。用户说"是"再动手。**

## 结束方式

探索没有固定的结束形式，可能是：

- 思路清晰了，用户自然转到其他事情
- 用户说"好，可以开始提案了"，顺势调用 propose 技能
- 用户说"先这样，稍后再说"，直接停下

如果思路已经收敛，可以选择性提供摘要：

```
## 探索小结

问题：<清晰化的问题描述>
方向：<如果已有共识>
待确认：<还有哪些开放问题>
```

这个摘要是可选的，有时思考本身就是价值。

## 注意事项

- 不写实现代码，如果用户要求实现，提醒他们先退出探索模式、创建变更提案
- 不假装理解，遇到不清楚的地方继续深挖
- 不催促，探索是思考时间不是执行时间
- 不强制结构化，让规律自然浮现
